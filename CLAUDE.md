# Project Memory

**同期ルール**: `AGENTS.md` は `CLAUDE.md` と基本同一内容に保つ。ただし「Codex 委譲ワークフロー」セクションは Claude Code 専用のため `AGENTS.md` には含めない。

## 開発セッション開始ルーティン

新しい開発セッションを開始する際は、**ドキュメントを最初に一括で読み込まない**。必要になった時にだけ、最小限を参照すること:

1. まずユーザーの依頼内容だけを確認する
2. 情報が不足している場合のみ、以下の順で**必要な範囲だけ**読み込む
   - `docs/CURRENT_TASK.md`（進行中タスクや TODO の確認が必要な場合）
   - `docs/CONVENTIONS.md`（実装・命名・禁止事項が必要な場合）
   - `docs/ARCHITECTURE.md`（構成や技術スタックが必要な場合）
3. ユーザーが明示的に「読む/確認する」と指示した場合は、その範囲を優先して参照する
4. 必要な情報が揃ったら、次に着手すべきタスクを提案する

## 言語ルール

- Codex の返答と更新するドキュメントは日本語で記入する

## Codex 委譲ワークフロー

ユーザーが「Codexに任せて」「Codexに委譲」「実装はCodexで」等と指示した場合、以下のプロトコルに従う:

### ステップ 1: 設計（Claude Code）
- 対象コードを読み、実装方針を設計する
- 設計内容を `docs/tasks/TASK-YYYYMMDD-NNN.md` に書き出す（テンプレートは `docs/tasks/_TEMPLATE.md`）
- ユーザーに設計を提示し、承認を得る

### ステップ 2: 委譲（Claude Code → Codex）
- MCP 経由で Codex に実装を指示する:
  ```
  mcp__codex__codex(
      prompt="docs/tasks/TASK-xxx.md を読み、記載された実装を行ってください。実装後は pytest を実行し、全テストがパスすることを確認してください。テストが失敗した場合は修正してから完了報告してください。",
      sandbox="danger-full-access",
      approval-policy="never"
  )
  ```
- Codex のレスポンスを確認し、必要なら `mcp__codex__codex-reply` で追加指示を送る

### ステップ 3: レビュー（Claude Code）
- `git diff` で Codex の変更内容を確認する
- Codex のテスト実行結果を確認し、必要に応じて再実行する
- 問題があれば修正し、問題なければユーザーに報告する
- タスクファイルのステータスを `completed` に更新する

### 常時委譲モード（`/codex-mode`）
- **このプロジェクトではデフォルトで常時委譲モードが有効**。セッション開始時から、明示的な指示がなくても全ての実装を Codex に委譲する
- ユーザーが `/codex-mode` を実行、または「委譲モードで」等と指示した場合も同様に動作する
- Claude Code は設計・レビューに専念し、コード変更は全て Codex 経由で行う
- ユーザーが「自分で実装して」と明示した場合のみ、Claude Code が直接実装する

### 注意事項
- Codex は `AGENTS.md` を自動で読み込むため、プロジェクト規約は引き継がれる
- タスクファイルは履歴として `docs/tasks/` に残す
- ステップ間でユーザーに進捗を報告すること

## タスク完了時ルーティン

タスクが完了したら、以下のドキュメントメンテナンスを行うこと:

1. **docs/CURRENT_TASK.md**: 完了したタスクにチェックを入れ、次のステップがあれば更新する
2. **docs/ARCHITECTURE.md**: 構成に変更があった場合（新しいライブラリ追加やファイル構成変更など）のみ更新する
3. **docs/MEMO.md**: 今回の実装内容、解決したエラー、重要な意思決定をログとして追記する
   - **必須**: 各エントリの冒頭に実行環境を `[claudecode]` または `[codex]` の形式で明示すること
