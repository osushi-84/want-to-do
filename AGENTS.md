# Project Memory

**同期ルール**: `AGENTS.md` は `CLAUDE.md` と基本同一内容に保つ。ただし「Codex 委譲ワークフロー」セクションは Claude Code 専用のため `AGENTS.md` には含めない。

## 開発セッション開始ルーティン

新しい開発セッションを開始する際は、**ドキュメントを最初に一括で読み込まない**。必要になった時にだけ、最小限を参照すること:

1. まずユーザーの依頼内容だけを確認する
2. 情報が不足している場合のみ、以下の順で**必要な範囲だけ**読み込む
   - `docs/CURRENT_TASK.md`（進行中タスクや TODO の確認が必要な場合）
   - `docs/CONVENTIONS.md`（実装・命名・禁止事項が必要な場合）
   - `docs/ARCHITECTURE.md`（構成や技術スタックが必要な場合）
3. ユーザーが明示的に「読む/確認する」と指示した場合は、その範囲を優先して参照する
4. 必要な情報が揃ったら、次に着手すべきタスクを提案する

## 言語ルール

- Codex の返答と更新するドキュメントは日本語で記入する

## テスト実行ルール

- 実装完了後は必ず `pytest` を実行し、全テストがパスすることを確認すること
- テストが失敗した場合は修正してから完了報告すること
- テスト結果（パス数・失敗数）を完了報告に含めること

## タスク完了時ルーティン

タスクが完了したら、以下のドキュメントメンテナンスを行うこと:

1. **docs/CURRENT_TASK.md**: 完了したタスクにチェックを入れ、次のステップがあれば更新する
2. **docs/ARCHITECTURE.md**: 構成に変更があった場合（新しいライブラリ追加やファイル構成変更など）のみ更新する
3. **docs/MEMO.md**: 今回の実装内容、解決したエラー、重要な意思決定をログとして追記する
   - **必須**: 各エントリの冒頭に実行環境を `[claudecode]` または `[codex]` の形式で明示すること
4. 実装作業が完了したら、関連変更を必ず `git commit` し、`git push` まで実施してから完了報告すること
